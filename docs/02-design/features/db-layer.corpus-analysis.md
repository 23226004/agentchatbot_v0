# db-layer 코퍼스 표본 분석 (Design 입력)

> **대상**: AI-Hub 법률 지식베이스 (`jikji_공부/project/Agent_00/tools/LAW_SEARCH_TOOL/data/.../01.법률`)
> **목적**: Fuseki/Qdrant 사이징·청크 추정·온톨로지 채택 판단. Design의 실측 근거.
> **측정일**: 2026-06-24 · **방법**: 스트리밍 집계(line/predicate/length)

> ## ⚠️ v2 정정 (실증 재검증, 2026-06-24)
> 독립 3관점 교차검증으로 아래 §2 주장 일부가 **틀린 것으로 판명**되어 정정한다.
> - ❌ **`citations`는 조문↔조문 상호참조가 아니다.** 객체 100%가 **판례 사건번호 리터럴**(예: "대법원 91므306"). 즉 조문→판례 엣지다. AI-Hub에 조문↔조문 참조 술어는 **없다**(상호참조는 본문 자연어로만 존재). → §2 "관계가 이미 다 있다"의 citations 항목 무효.
> - ❌ **AI-Hub Article `fullText`=조 제목만**, 본문은 Paragraph 노드에. 게다가 `hasParagraph` 링크가 URI 공백 불일치로 **100% 단절**. → AI-Hub를 본문 소스로 못 씀.
> - ❌ **건축법·"거실"이 AI-Hub에 없음**, law.go.kr nt도 건축법 본문 0.
> - ✅ **`상위법령`(위임) 13,173엣지, 라벨 100% 보유** — `delegatesTo`는 공짜(생성 불요).
> - **결론 → 피벗**: 본문은 **law.go.kr 실시간 API**가 유일 소스. AI-Hub는 **관계그래프(위임)** 용으로만. 상세는 아래 §6, Design v0.3.

## 2. 핵심 발견 — **관계가 이미 다 들어있다** (regex 파싱 불필요)

Plan에서 "상호참조를 정규식으로 파싱"하려던 것이 **이미 트리플로 존재**:

| 술어 | 건수 | 의미 (= Plan의 어떤 엣지) |
|------|-----:|------|
| `kb/law/citations` | 19,877 (found) | **조문↔조문 상호참조** (REFERENCES) |
| `kb/law/hasParagraph` | 8,868 (found) | 조문→항 (HAS_PARAGRAPH) |
| `kb/law/containingWord` | 20,682 (found) | 조문→용어 (DEFINES/uses) |
| `kb/law/refArticle` | **206,127** (case) | **판례→조문** |
| `kb/law/refPrecedent` | 107,560 (case) | 판례→판례 |
| `kb/law/related` | **1,302,860** (term) | 용어↔용어 (+ synonym/hypernym/antonym) |

→ **FR-06(관계확장)의 엣지를 만들 필요 없이 그대로 적재하면 됨.** 위임(delegatesTo)만 law.go.kr 메타로 보강.

| 종류 | 건수 | 평균 | 중앙값 | p90 | 최대 |

## 4. 사이징 결론

**Fuseki (TDB2)**
- 4.84M 트리플 → 디스크 ~1.5–2.5GB 추정. `tdb2.tdbloader` 벌크 적재 **수 분** (SPARQL UPDATE 적재는 느리니 벌크 사용).

**Qdrant (BGE-m3 dense 1024-d + sparse)**
- 이 코퍼스: 법령 ~17K + 판례 분할 ~80–200K → **~100–220K 벡터**. dense 1024×4B ≈ 0.4–0.9GB + sparse + payload. **여유**.
- 현행 전체(law.go.kr, ~5,400 법령 추정 ~200–300K 조문)로 확장해도 수십만~백만 벡터 → Qdrant 정상 범위.

**임베딩 시간 (4090, BGE-m3)**
- 이 코퍼스 ~100–220K 청크 → 처리율 ~100–200/s 가정 시 **~10–40분 1회 배치**. 현행 전체도 동일 자릿수.

## 5. Design 반영 사항

1. **어휘 채택**: `aihub.or.kr/kb/law` 온톨로지를 **base vocabulary로 채택**(이미 정합)하고 law.go.kr 술어를 매핑 + ELI(버전) 추가. → Plan §7.2 "매핑" 구체화.
2. **관계확장**: citations/refArticle/refPrecedent/related를 그대로 그래프 엣지로. 정규식 파싱 제거.
3. **청크 단위**: 법령=조(Article), 판례=분할(슬라이딩/문단). 이상치 길이 분할 규칙 필요.
4. **스코프 결정 필요**: 판례(1.5M 트리플·77K 본문)를 1차 포함 vs 2차. 법령+판례면 검색 표면이 크게 넓어짐.
5. **신선도**: 이 코퍼스는 구조·관계 부트스트랩. 본문·시행일은 law.go.kr 현행으로 갱신(Plan §1.4).

## 6. 조인 정합성 실증 (v0.3, "엉뚱한 연결" 방지)

본문을 law.go.kr API로, 관계를 AI-Hub로 가져올 때 **두 소스를 어떻게 안전하게 붙이는가**를 실데이터로 검증.

**law.go.kr-nt 법령 노드 구조**
- URI = `법령/_{내부ID}-{시행일자}` (예: `_191693-20170207`), `rdfs:label`=깨끗한 법령명.
- **내부ID는 API 법령ID/MST와 다른 체계** → 조인키로 부적합. 라벨(법령명)이 유일한 연결 후보.
- `상위법령` 위임엣지 13,173건, 양끝 라벨 보유율 100%.

**API 재해소 모호도 측정** (표본 18개 법령명 → `lawSearch` 정확명+현행)
| 결과 | 건수 |
|------|-----|
| ✅ 유일 법령ID 해소 | 17/18 (94%) |
| ⚠️ 복수(모호) | **0** |
| ❌ 무일치 | 1 (상이기장령→헌신영예기장령, 개정 명칭변경) |

**검증된 조인 전략**
1. AI-Hub 내부ID 무시. **조인키 = 법령명 정확일치 + "현행" → 유일 법령ID**(모호 0 실측).
2. **본문·조문 정체성 = law.go.kr API 단일 소스**(법령ID+시행일자+조문번호). 교차조인 없음 → 본질적 자기정합.
3. **위임그래프**: `상위법령` 양끝 법령명을 각각 API 재해소, **둘 다 유일ID로 풀릴 때만** `delegatesTo` 엣지 생성.
4. **해소 실패(개정·폐지·조례) = 격리(quarantine)+로그**, 강제연결 금지. ← 엉뚱한 연결 차단의 핵심.
5. **인제스천 검증 게이트**: 해소 리포트(해소/모호/격리 건수) 산출, **모호=0 확인 후** 엣지 활성화.

**위임 커버리지 정정 (80표본 재검증)**: 상위법령 13,173엣지 중 **국가법령→법령은 534개**(96%는 자치법규(조례), `/자치법규/` 네임스페이스로 제외). 80표본 해소: 유일 91.3%·**모호 0**·격리 8.8%(2018↔현행 명칭변경: 문화재청→국가유산청 등). 양끝 생존 ~83% → **실효 위임엣지 ≈ 444개**. "13K 위임"은 조례 포함 오해이며 1차 실효는 ~444. 모호 0은 표본 확대에도 견고.
